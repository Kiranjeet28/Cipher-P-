Abstract

This project presents CipherVault, a complete image encryption and decryption engine that applies classical and stream cryptographic algorithms directly to the pixel-level byte representation of digital images. The system is built on a modular C++17 backend that exposes a RESTful API on port 5000. A modern, animated browser-based frontend communicates with this backend via HTTP multipart requests, submitting raw image data and receiving a cryptographically transformed PNG in response.

The image encryption pipeline operates as follows: images are decoded into RGBA pixel buffers, the RGB channels are extracted into a linear byte stream, the selected cipher algorithm transforms each byte through its mathematical function, and the modified byte stream is reassembled into a valid PNG output. Seven algorithms are implemented: Caesar (additive shift), Multiplicative, Affine (linear transform), Playfair (16×16 digraph substitution), Hill (matrix multiplication mod 256), RC4 (stream cipher with KSA/PRGA), and Rail Fence (transposition). Decryption is the mathematical inverse of each operation, verified to perfectly reconstruct the original image.

The frontend features animated encryption visualizations, real-time pipeline state indicators, a backend health monitor, drag-and-drop image upload, before/after comparison canvases, and a download facility. All processing is performed by the C++ engine — the browser performs no cryptographic computation.

1. Introduction

Digital images are among the most transmitted forms of data in modern communication systems. Whether in medical imaging, surveillance, personal communication, or cloud storage, the protection of visual information against unauthorised access is a critical engineering concern. Image encryption transforms a visually meaningful photograph into a perceptually random pattern of pixels by applying a reversible mathematical function to the underlying byte representation of the image's colour channels.

This project, CipherVault, implements a complete image encryption and decryption system that operates at the pixel byte level. Rather than encrypting image files as opaque binary streams (as a file encryption tool would), CipherVault decodes each image into its RGBA pixel buffer, extracts the three colour channels (R, G, B) into a linear byte array, applies a cryptographic transformation to every byte, and reconstructs a valid PNG image from the transformed channel data. This approach produces images that remain structurally valid — the encrypted output is a displayable PNG — while containing visually scrambled pixel data.

Seven cryptographic algorithms are implemented: three classical substitution ciphers (Caesar, Multiplicative, Affine), one digraph substitution cipher (Playfair), one matrix-based cipher (Hill), one stream cipher (RC4), and one transposition cipher (Rail Fence). All algorithms operate on the RGB byte stream of the image with modulus 256 to remain within the valid byte range.

3. Image Processing Engine

The image processing engine is the foundation upon which all cryptographic transformations are built. It translates between the opaque binary representation of an image file and the structured, addressable memory layout that algorithms require. Understanding this translation is essential for understanding why image encryption produces visually scrambled but structurally valid output.

3.1 Image Loading and Decoding

When a user submits an image, the frontend serialises the canvas content as a PNG blob and attaches it to a FormData object. The backend's ImageLoader::loadFromMemory() receives the raw binary content and calls stbi_load_from_memory() with the STBI_rgb_alpha flag, which forces four-channel (RGBA) output regardless of the source format. This normalisation ensures that PNG, JPEG, and WebP inputs produce identically structured buffers.

Why force RGBA? JPEG images have 3 channels; PNG may have 1, 2, 3, or 4. Normalising to RGBA at load time means the encryption engine always sees a predictable 4-byte-per-pixel layout. The alpha channel is preserved unchanged across all operations.

The result is an ImageData structure containing the decoded pixel buffer as a contiguous std::vector<unsigned char> of size width × height × 4 bytes:

struct ImageData { int width, height, channels; // channels = 4 (RGBA) std::vector<unsigned char> pixels; // size = width * height * 4 };

3.2 Pixel Extraction and Memory Layout

Pixels are stored in row-major order. For an image of width W and height H, the byte at position i within a row y and column x is addressed as:

offset(x, y) = (y × W + x) × 4 R = pixels[offset], G = pixels[offset+1], B = pixels[offset+2], A = pixels[offset+3]

PixelProcessor::extractRGBStream() traverses every pixel in row-major order and produces a linear byte vector containing only the R, G, B bytes — the alpha channel is excluded from transformation:

// From PixelProcessor.cpp for (size_t i = 0; i < pixelCount; ++i) { size_t base = i * img.channels; // channels = 4 stream.push_back(img.pixels[base+0]); // R stream.push_back(img.pixels[base+1]); // G stream.push_back(img.pixels[base+2]); // B // alpha (base+3) intentionally excluded }

For a 1920×1080 image, this produces a byte stream of 1920 × 1080 × 3 = 6,220,800 bytes. This is the exact input to the algorithm engine.

3.3 RGB Byte Stream Generation

The stream is a std::vector<unsigned char> where values are in the range [0, 255]. Each algorithm receives this vector and returns a transformed vector of equal size. The mathematical operations are all defined modulo 256, ensuring that the output values remain valid byte values and can be directly written back as pixel colours.

Some algorithms (Playfair, Hill) operate on pairs or blocks of bytes rather than individual bytes. For these, the stream may be padded with trailing zero bytes to reach a multiple of the block size. This padding is accounted for during decryption by trimming the stream back to its original length before reconstruction.

3.4 Image Reconstruction and Encoding

After encryption, PixelProcessor::reconstructRGBStream() writes the transformed bytes back into the RGBA pixel buffer, placing each byte at its original R/G/B position while leaving the alpha channel at its original value. ImageReconstructor::encodePNG() then calls stbi_write_png_to_func() with a callback that appends bytes to an output vector, which is streamed directly to the HTTP response via res.set_content_provider(). No temporary files are written to disk at any stage.

5. Algorithm Implementations

5.1 Caesar Cipher

Nature

The Caesar cipher is a monoalphabetic additive substitution cipher. When applied to image bytes, each channel value is shifted by a constant integer k in modular arithmetic over Z256. Because the shift is uniform across all pixels and channels, the statistical distribution of values is preserved — a histogram attack trivially recovers the key.

Mathematical Operation

(5.1) Encryption: C = (P + k) mod 256 Decryption: P = (C − k) mod 256 Key space: k ∈ {0, 1, …, 255}

Pixel Transformation Example

B = 200

Encrypted Pixel

G = 71
B = 207

Security on Images

Caesar encryption on images is visually ineffective — the output image retains structural outlines because the relative differences between neighbouring pixels are unchanged. Statistical attacks recover k in at most 255 trials. It serves as a baseline demonstration only.

5.2 Multiplicative Cipher

Nature

The multiplicative cipher replaces each byte value by its product with a key a in Z256. For decryption to be possible, a must be invertible modulo 256, which requires that gcd(a, 256) = 1. Because 256 = 28, the key must be an odd integer. The backend validates this using ModularArithmetic::isCoprime(a, 256) before processing begins.

Mathematical Operation

(5.2) Encryption: C = (a × P) mod 256 Decryption: P = (a⁻¹ × C) mod 256 where a⁻¹ is found via Extended Euclidean Algorithm: a⁻¹ such that a × a⁻¹ ≡ 1 (mod 256)

The decryption key for a = 5 is a⁻¹ = 205, since 5 × 205 = 1025 ≡ 1 (mod 256). The backend computes this automatically via the Extended Euclidean Algorithm in ModularArithmetic::modInverse().

5.3 Affine Cipher

Nature

The affine cipher generalises both Caesar (b ≠ 0, a = 1) and multiplicative (a ≠ 1, b = 0) into a single linear transformation over Z256. Two keys are required: a multiplicative component a (must be coprime to 256) and an additive component b. The frontend accepts these as a comma-separated pair, e.g. 5,8.

Mathematical Operation

(5.3) Encryption: C = (a × P + b) mod 256 Decryption: P = a⁻¹ × (C − b) mod 256 Condition: gcd(a, 256) = 1

5.4 Playfair Cipher (16×16 Extended)

Nature

The classical Playfair cipher uses a 5×5 grid for 25 characters. This implementation extends the concept to a 16×16 grid of all 256 byte values (0–255), enabling it to operate on arbitrary binary data including image bytes. Byte pairs are processed using the same three positional rules as the original cipher: same row (shift right/left), same column (shift down/up), or rectangle swap.

Key Square Construction

The 16×16 substitution square is constructed from the key string by inserting key bytes first (avoiding duplicates), then filling remaining positions with all unused byte values 0–255. The key can be a plain string or a hexadecimal string of even length. This deterministic construction means the same key always produces the same square, enabling inverse operations on decryption.

Digraph Processing Rules

For byte pair (P₁, P₂) at grid positions (r₁,c₁) and (r₂,c₂): Same row (r₁ = r₂): C₁ = square[r₁][(c₁+1) mod 16], C₂ = square[r₂][(c₂+1) mod 16] Same col (c₁ = c₂): C₁ = square[(r₁+1) mod 16][c₁], C₂ = square[(r₂+1) mod 16][c₂] Rectangle: C₁ = square[r₁][c₂], C₂ = square[r₂][c₁]

Decryption uses shift −1 (i.e., +15 mod 16) for row and column rules, and the rectangle rule is its own inverse.

5.5 Hill Cipher

Nature

The Hill cipher applies a linear algebraic transformation over Z256. An n×n key matrix is multiplied by a column vector of n plaintext bytes to produce a ciphertext vector. This implementation supports 2×2 and 3×3 matrices. The key requirement is that the matrix determinant must be invertible modulo 256, i.e., gcd(det(K), 256) = 1.

Mathematical Operation

(5.5) Encryption: C = K · P (mod 256) Decryption: P = K⁻¹ · C (mod 256) where K⁻¹ = det(K)⁻¹ · adj(K) (mod 256)
2×2 example with K = [[3,2],[5,7]]: det(K) = 3·7 − 2·5 = 11, det(K)⁻¹ mod 256 = 163 (since 11·163 ≡ 1 mod 256) adj(K) = [[7,−2],[−5,3]], K⁻¹ = 163·[[7,254],[251,3]] mod 256

Block Processing on Image Bytes

The RGB byte stream is divided into blocks of size n (2 or 3). Each block is multiplied by the key matrix modulo 256. Since the RGB stream length is 3 × (number of pixels), which is always divisible by 3, no padding is needed for 3×3 matrices. For 2×2 matrices, a single zero byte may be appended if the stream length is odd. This byte is stripped during decryption.

Input block (n=2)

5.6 RC4 Stream Cipher

Nature

RC4 is a synchronous stream cipher that generates a pseudo-random byte stream (keystream) from the key, then XORs this stream with the plaintext. Because XOR is its own inverse and the keystream is deterministic given the key, encryption and decryption are identical operations. RC4 is particularly suitable for image encryption because it operates on individual bytes and does not require padding.

Key Scheduling Algorithm (KSA)

The KSA initialises a 256-byte substitution array S with the identity permutation, then scrambles it using the key bytes:

S[i] = i for i in 0..255 j = 0 for i in 0..255: j = (j + S[i] + key[i mod len(key)]) mod 256 swap(S[i], S[j])

Pseudo-Random Generation Algorithm (PRGA)

The PRGA generates one keystream byte per input byte:

i = 0, j = 0 for each input byte P[k]: i = (i + 1) mod 256 j = (j + S[i]) mod 256 swap(S[i], S[j]) K[k] = S[(S[i] + S[j]) mod 256] // keystream byte C[k] = P[k] XOR K[k] // encrypted byte

Mathematical Operation

(5.6) Encryption: C[k] = P[k] ⊕ KeyStream[k] Decryption: P[k] = C[k] ⊕ KeyStream[k] (identical — XOR is self-inverse)

Pixel Transformation Example (key = "secret")

The first keystream byte produced by the KSA/PRGA with key "secret" is deterministic. For the pixel R=120: if the first keystream byte is 0xAE = 174, then C = 120 XOR 174 = 214. The original value is recovered by 214 XOR 174 = 120.

5.7 Rail Fence Cipher

Nature

Rail Fence is a transposition cipher — it rearranges bytes without changing their values. The bytes are written in a zigzag pattern across r rows ("rails"), then read off row by row to produce the ciphertext. Unlike substitution ciphers, the statistical distribution of byte values is unchanged; the cipher's effect is purely positional reordering. Its visual effect on images is a characteristic striping distortion.

Zigzag Transposition

For a sequence of bytes B[0..n-1] and r rails, each byte is assigned to rail rail(i) following a bouncing pattern: 0, 1, …, r-1, r-2, …, 1, 0, 1, … The ciphertext is formed by concatenating all bytes on rail 0, then rail 1, …, rail r−1.

(5.7) For r=3 and input [B₀ B₁ B₂ B₃ B₄ B₅ B₆ B₇ B₈ ...]: Rail 0: B₀ B₄ B₈ ... Rail 1: B₁ B₃ B₅ B₇ ... Rail 2: B₂ B₆ ... Ciphertext = [Rail 0 bytes] + [Rail 1 bytes] + [Rail 2 bytes]

Decryption requires computing the rail index for each position in the ciphertext and reversing the interleaving. The backend implementation in RailFenceCipher::decryptRailFence() precomputes the rail assignment for all positions, builds offset arrays per rail, then reassembles the original sequence in one pass.

6. Mathematical Analysis

6.1 Extended Euclidean Algorithm

The core mathematical primitive used by Caesar decryption, Multiplicative, Affine, and Hill ciphers is the modular multiplicative inverse, computed by the Extended Euclidean Algorithm. The backend's ModularArithmetic::extendedGCD(a, b) and modInverse(a, m) implement this directly.

(6.1) Extended GCD: Find x, y such that a·x + b·y = gcd(a, b) If gcd(a, m) = 1, then x is the modular inverse of a modulo m: a·x ≡ 1 (mod m)
Example: Find 5⁻¹ mod 256 gcd(5, 256): 256 = 51×5 + 1 → 1 = 256 − 51×5 Therefore x = −51 ≡ 205 (mod 256) Verification: 5 × 205 = 1025 = 4×256 + 1 ≡ 1 (mod 256) ✓

6.2 Hill Cipher Matrix Inversion

For a 2×2 key matrix K, the modular matrix inverse is:

(6.2) K = [[a, b], [c, d]] det(K) = a·d − b·c adj(K) = [[d, −b], [−c, a]] K⁻¹ = det(K)⁻¹ · adj(K) (mod 256)
Worked example: K = [[3, 2], [5, 7]] det(K) = 21 − 10 = 11 11⁻¹ mod 256 = 163 (since 11 × 163 = 1793 = 7×256 + 1) adj(K) mod 256 = [[7, 254], [251, 3]] (−2 mod 256 = 254, −5 mod 256 = 251) K⁻¹ = 163 × [[7, 254], [251, 3]] mod 256 = [[141, 50], [21, 233]]

6.3 RC4 Key Space

The key space of RC4 is determined by the key length. For a key of l bytes (l = 1..256), the number of distinct keys is 256l. Even at l = 8 bytes, the key space is 2568 = 264 ≈ 1.8 × 1019, which is computationally infeasible to brute-force. The primary practical weakness of RC4 is key reuse and statistical biases in the early keystream, which do not apply in the batch image encryption context used here.

6.4 Modular Arithmetic Over Z₂₅₆ for Image Processing

All ciphers operate over Z256 because image channel values are unsigned 8-bit integers in the range [0, 255]. The C++ implementation uses bitwise AND to enforce this bound efficiently:

result = (a × b + c) & 0xFF ≡ (a × b + c) mod 256

This works because 256 = 28 and masking with 0xFF = 28 − 1 is equivalent to computing mod 28. For subtraction, the two's complement wrap-around is corrected as (a − b) & 0xFF, which produces the correct non-negative residue.

8.2 Specific Vulnerabilities

Caesar and Multiplicative ciphers applied to images are vulnerable to histogram analysis: because each unique input value maps to exactly one output value, the histogram shape is preserved. An attacker can recover the key by comparing histograms.

Hill cipher is vulnerable to known-plaintext attacks if the attacker has access to a plaintext-ciphertext pair of length ≥ n bytes: solving the system K · P = C mod 256 for K requires only matrix inversion.

RC4 has known statistical weaknesses in the first few bytes of the keystream (Fluhrer–Mantin–Shamir, 2001). For image encryption, discarding the first 256 bytes of keystream (RC4-drop[256]) would mitigate this, though this optimisation is not implemented in the current version.

Rail Fence provides no confidentiality against statistical analysis — only the positions of bytes change, not their values.

8.3 System-Level Security

The backend performs key validation before any encryption begins. For Multiplicative and Affine ciphers, isCoprime(key, 256) is checked, and an HTTP 400 response is returned if the key is invalid. For Hill cipher, matrix invertibility is verified by computing gcd(det(K), 256) = 1 before encryption. These checks prevent silent data corruption from uninvertible keys.

CORS headers are set on all responses, allowing the frontend (served from any origin including file://) to receive the encrypted image. The backend does not validate request origins, which is appropriate for a local development system.

10. Limitations and Future Scope

10.1 Current Limitations

  • Single-threaded HTTP server — the cpp-httplib server processes one request at a time. For large images or concurrent users, a thread-pool configuration would be needed.
  • Classical cipher security — Caesar, Multiplicative, Affine, and Rail Fence ciphers are academically demonstrative but cryptographically weak. They are included for educational comparison, not secure communication.
  • RC4 early bias — the implementation does not drop the first 256 keystream bytes (RC4-drop[256]), leaving it susceptible to Fluhrer–Mantin–Shamir statistical attacks on the keystream head.
  • No authentication — there is no MAC or authenticated encryption. An attacker with access to the encrypted image can modify it without detection.
  • 16-bit pixel depth not supported — images with 16-bit channel depth (medical DICOM, high-dynamic-range) are not handled; stb_image decodes them to 8-bit.

10.2 Future Scope

  • AES-256 in CBC/GCM mode — replacing or supplementing RC4 with AES would provide industry-standard security with authenticated encryption.
  • Frequency analysis visualisation — displaying RGB histograms before and after encryption would visually demonstrate the statistical impact of each cipher on the image data.
  • Entropy calculation — computing Shannon entropy H(P) = −Σ p(x) log₂ p(x) on the pixel value distribution would quantify the randomness introduced by each cipher.
  • Multi-image batch processing — extending the API to accept archives of images and process them in parallel using a thread pool.
  • Steganography integration — embedding a watermark or hidden message within the alpha channel of the encrypted image.

11. Conclusion

This project demonstrates a complete, functionally verified image encryption and decryption system implemented as a full-stack application. The C++17 backend implements seven cryptographic algorithms operating at the byte level of image data, connected through a modular pipeline that separates image loading, pixel processing, algorithm execution, and image reconstruction into independent, reusable components. All algorithms produce losslessly reversible encrypted images — decryption with the correct key exactly recovers the original image.

The frontend provides a professional animated interface that visualises the real backend processing pipeline rather than simulating it. Every encryption operation travels from the browser to the C++ engine and back as a real HTTP transaction, making the system a genuine full-stack cryptographic application rather than a client-side demonstration.

The modular architecture makes the system extensible: adding a new cipher requires only implementing the four-method CipherBase interface and registering the new class in CipherFactory. The image processing pipeline — decode, extract, transform, reconstruct, encode — remains unchanged regardless of which algorithm is selected.

The project provides a practical platform for studying how classical cryptographic algorithms behave when applied to image data, demonstrating the trade-offs between key complexity, visual encryption quality, and cryptographic security within a unified, interactive system.
