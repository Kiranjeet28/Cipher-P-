1. Introduction

Cryptography protects data by transforming plaintext into ciphertext using a key and a reversible algorithm. When applied to images, the data is processed as bytes or pixels, and the output must still reconstruct a valid image. Image encryption is important for medical imaging, surveillance, personal media, and secure communication.

2. Project Objective

The objective is to analyze how multiple cryptographic algorithms behave internally on image data, show their mathematical operations, and demonstrate how byte-level or matrix-level transformations alter the visible image. The project also compares the algorithms by security, complexity, and practical relevance.

5.2 Pixel Extraction and Memory Layout

Pixels are stored row by row in memory. Each pixel contains three color bytes, usually interpreted as R, G, and B, while alpha may be preserved separately. The extraction step reads each pixel from the decoded image buffer and appends its channels into a linear sequence.

PixelRGB valuesFlattened formByte stream

flowchart TB A[Pixel (R,G,B)] --> B[R = 120] B --> C[G = 64] C --> D[B = 200] D --> E[[120, 64, 200]] E --> F[BYTE STREAM]
Figure 3: Pixel-to-byte transformation

The layout is row-major in memory: pixels are read left to right across a row, then continued on the next row. This means the encryption engine operates on a linear stream even though the image is two-dimensional.

5.3 Image-to-Byte Conversion

The decoded image is transformed into a mathematical data structure. The 2D pixel matrix becomes a serialized byte vector. This flattening is essential because cryptographic algorithms work on numbers, not on visual shapes. The conversion process preserves ordering so that the encrypted stream can later be reconstructed into image pixels.

Pixel matrix → serialized RGB vector → encryption input buffer → transformed ciphertext vector → reconstructed pixel matrix.

6. Mathematical Analysis & Working Nature

This section gives the internal mathematical behavior of each algorithm, including stepwise byte calculations, pixel transformation examples, key derivation behavior, and the nature of the cipher.

6.1 Caesar Cipher

Nature: Additive substitution cipher.

Encryption: \(C_i = (P_i + k) \bmod 256\)

Decryption: \(P_i = (C_i - k) \bmod 256\)

Working nature: Each byte is shifted independently. The algorithm does not mix neighboring pixels, so image structure is still partly visible.

Input ByteKeyOperationOutput Byte
1203(120 + 3) mod 256123
25510(255 + 10) mod 2569
05(0 + 5) mod 2565

Pixel example: RGB (120, 200, 45) with k=7 becomes (127, 207, 52).

flowchart TB A[Input bytes] --> B[Add key k] B --> C[mod 256] C --> D[Ciphertext bytes]
Figure 2: Caesar internal workflow

6.2 Multiplicative Cipher

Nature: Multiplicative substitution cipher.

Encryption: \(C_i = (aP_i) \bmod 256\), with \(\gcd(a,256)=1\)

Decryption: \(P_i = a^{-1}C_i \bmod 256\)

Inverse calculation: find \(a^{-1}\) such that \(aa^{-1}\equiv1\pmod{256}\). For example, \(5^{-1}\equiv205\) because \(5\cdot205=1025\equiv1\pmod{256}\).

Input ByteKeyOperationOutput Byte
45(5 × 4) mod 25620
20205(205 × 20) mod 2564

Pixel example: RGB (12, 34, 56) with a=5 becomes (60, 170, 24).

flowchart TB A[Plain bytes] --> B[Multiply by a] B --> C[mod 256] C --> D[Ciphertext] D --> E[Multiply by a^-1]
Figure 3: Multiplicative cipher workflow

6.3 Affine Cipher

Nature: Linear substitution cipher combining multiplication and addition.

Encryption: \(C_i = (aP_i + b) \bmod 256\)

Decryption: \(P_i = a^{-1}(C_i - b) \bmod 256\)

Derivation: Starting from \(C = aP + b\), subtract \(b\) and multiply by \(a^{-1}\) to recover plaintext.

Input ByteKeyOperationOutput Byte
10a=5,b=8(5 × 10 + 8) mod 25658
58a^-1=205,b=8205 × (58 - 8) mod 25610

Pixel example: RGB (20, 40, 60), a=5, b=12 gives (112, 212, 56).

flowchart TB A[Input bytes] --> B[Multiply by a] B --> C[Add b] C --> D[mod 256] D --> E[Ciphertext]
Figure 4: Affine cipher workflow

6.4 Playfair Cipher

Nature: Digraph substitution cipher adapted here to a 16×16 byte square.

Working nature: Two bytes are processed together. A 256-byte key square is generated from the seed key, then pairwise rules are applied depending on row and column positions.

Rules: Same row shifts right, same column shifts down, otherwise rectangle corners are swapped.

Pair InputRuleResult
(a,b)Same rowShift both right by 1
(x,y)Same columnShift both down by 1
(p,q)RectangleSwap columns

Pixel example: RGB bytes are grouped into pairs, so the visible effect is pairwise scrambling of color bytes while preserving overall byte count.

flowchart TB A[Input byte stream] --> B[Build 16x16 key square] B --> C[Process byte pairs] C --> D[Row/Column/Rectangle rule] D --> E[Ciphertext pairs]
Figure 5: Playfair byte-pair processing

6.5 Hill Cipher

Nature: Matrix-based block cipher.

Encryption: \(\mathbf{C}=K\mathbf{P}\bmod256\)

Decryption: \(\mathbf{P}=K^{-1}\mathbf{C}\bmod256\)

Matrix inverse: for \(K=\begin{pmatrix}a&b\\c&d\end{pmatrix}\), \(\det K=ad-bc\). If \(\gcd(\det K,256)=1\), then \(K^{-1}=(\det K)^{-1}\operatorname{adj}(K)\bmod256\).

Plain VectorKey MatrixOperationCipher Vector
[10, 20]^T[[3, 3],[2, 5]][3×10+3×20, 2×10+5×20] mod 256[90, 120]^T

Pixel example: adjacent RGB bytes are grouped into blocks, so one changed byte affects multiple outputs. This creates better diffusion than simple substitution.

flowchart TB A[Input blocks] --> B[Key matrix K] B --> C[Multiply K × P] C --> D[mod 256] D --> E[Ciphertext blocks]
Figure 6: Hill matrix workflow

6.6 Rail Fence Cipher

Nature: Transposition cipher.

Working nature: The byte values are not changed, only their positions are permuted in a zig-zag rail pattern. This preserves the histogram but scrambles the spatial order of pixels.

IndexOperationOutput
0..NAssign by zig-zag railsPermuted stream

Pixel behavior: image structure becomes visually scrambled while byte distribution remains unchanged.

6.7 Diffie-Hellman Key Exchange

Nature: Asymmetric key exchange protocol, not a direct image cipher.

Equations: \(A=g^a\bmod p\), \(B=g^b\bmod p\), shared key \(s=B^a\bmod p=A^b\bmod p=g^{ab}\bmod p\).

Worked example: with \(p=23\), \(g=5\), \(a=6\), \(b=15\): \(A=8\), \(B=19\), shared secret \(s=2\).

Working nature: The protocol derives a shared symmetric key, which can then be used to encrypt image bytes with a symmetric cipher.

flowchart TB A[Choose p,g] --> B[Compute public values] B --> C[Exchange A and B] C --> D[Compute shared secret] D --> E[Use shared key for image encryption]
Figure 8: Diffie-Hellman workflow

6.8 RSA Algorithm

Nature: Asymmetric public-key algorithm.

Key generation: choose primes \(p,q\), compute \(n=pq\), \(\phi(n)=(p-1)(q-1)\), choose \(e\), compute \(d\equiv e^{-1}\pmod{\phi(n)}\).

Encryption: \(C=M^e\bmod n\)

Decryption: \(M=C^d\bmod n\)

Worked example: \(p=61,q=53\Rightarrow n=3233,\phi=3120\). With \(e=17\), the modular inverse is \(d=2753\). For \(M=65\), encryption gives \(C=2790\), and decryption restores \(65\).

MessageKeyOperationOutput
65e=1765^17 mod 32332790
2790d=27532790^2753 mod 323365

Working nature: RSA is better for key encapsulation than encrypting whole images directly.

flowchart TB A[Prime selection] --> B[Compute n and phi] B --> C[Choose e] C --> D[Compute d] D --> E[Encrypt key or small message]
Figure 9: RSA key and encryption workflow

6.9 RC4 Stream Cipher

Nature: Stream cipher based on XOR keystream generation.

KSA: initialize \(S[0..255]\), then permute using the key. PRGA: produce keystream bytes and XOR them with plaintext bytes.

Encryption / decryption: \(C_i = P_i \oplus K_i\), and because XOR is self-inverse, \(P_i = C_i \oplus K_i\).

Plain ByteKeystream ByteOperationCipher Byte
17095170 XOR 95245
24595245 XOR 95170

Working nature: RC4 changes byte values through XOR, producing strong byte scrambling if the keystream is unique. However, practical biases and key reuse make it unsuitable for modern secure systems.

7. Internal Workflow Nature and Image Transformation

Image bytes enter the algorithm as RGB channel values. Depending on the cipher, each byte may be shifted, multiplied, substituted, permuted, mixed by a matrix, or XORed with a keystream. The transformed stream is then reassembled into an image. Algorithms with only substitution preserve some patterns; transposition preserves byte values but moves positions; matrix and XOR-based ciphers create stronger visual disruption.

9. Image Processing Section

The backend decodes PNG into RGBA, processes only RGB bytes, preserves alpha, and writes the transformed buffer back to PNG. In mathematical terms, pixels are treated as byte vectors. For substitution ciphers, each channel changes independently. For Hill, bytes are mixed within blocks. For RC4, bytes are XORed with a keystream. The resulting image often shows altered colors, noise-like scrambling, or scrambled spatial order depending on the algorithm nature.

12. Advantages

The project compares multiple cipher families, shows real image byte behavior, and provides a full-stack demonstration of image encryption and decryption using a modular architecture.

13. Limitations

Classical ciphers are not secure for production use. Some algorithms are conceptual or educational in nature. Hill is limited to small matrices in the current implementation, and secure asymmetric workflows would require a dedicated big-number library.

14. Future Scope

Future work should include authenticated encryption such as AES-GCM, secure key exchange with real RSA/DH libraries, stronger image-dedicated cryptosystems, and automated verification of reversibility and security.

15. Conclusion

This report demonstrates the mathematical behavior of diverse cryptographic algorithms on image data. It highlights how byte transformations, transpositions, matrix operations, and keystream XOR affect the visual and security properties of encrypted images.
